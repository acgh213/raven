## 1. Executive Verdict

**Raven has a stronger foundation than its feature surface suggests — and that's exactly right for 11 commits.** The project's central thesis ("ranking and annotation, never deletion") is reinforced at every layer: schema, ADRs, non-goals, code structure, and operational contracts. The engineering discipline is unusually high for a project this early — the job queue's deduplication with race-condition recovery, the SQLite connection management with documented rationale, and the test coverage of edge cases (expired leases, wrong leases, deferred FKs, empty-file backup refusal) are all signs of someone who's been burned by production incidents before and is building accordingly. 

The primary risk is not quality but velocity: ~20% of v0.1 is built, and the remaining surface (SSRF-safe fetcher, extractor, feed/article APIs, sync protocol, Android client) is substantial. The one **critical finding** is that bearer-token auth — promised by the design docs — is parsed from config but never wired into the HTTP layer. That's a P0 that must be fixed before any non-healthz endpoint ships.

## 4. Findings

### 🔴 P0 — Blocks trustworthy release or security boundary

**P0-1: Bearer-token auth middleware is missing**

`Config.APIToken` is parsed from `RAVEN_API_TOKEN` env var (`config.go:75`) but **never wired into the HTTP layer**. `app.New()` takes no parameters and `cmd/raven/main.go` doesn't pass the config to app construction. The design explicitly requires: *"Bearer-token auth applies even on Tailscale; a small unauthenticated local health check is allowed only when explicitly configured."*

- **Files:** `cmd/raven/main.go:62`, `internal/app/app.go:13-20`
- **Severity:** Any non-healthz endpoint added before auth middleware will ship unauthenticated.
- **Fix:** Add bearer-token auth middleware to `internal/app/` that wraps all non-healthz routes. Pass `config.APIToken` through app construction. Add a config flag (`RAVEN_ALLOW_UNAUTHENTICATED_HEALTHZ`) for the explicit healthz toggle the design requires.

**P0-2: SSRF-safe fetching is entirely unimplemented**

DESIGN.md makes strong promises about URL safety: *"Every feed and article URL is untrusted input. Raven only fetches HTTP(S), rejects URL credentials, follows a limited redirect chain, and blocks loopback, link-local, private, Tailscale, and metadata-service targets before and after DNS/redirect resolution."* None of this exists — there's no `internal/fetcher/` directory at all. The worker handler map is empty (`main.go:58`), so even if feed jobs were enqueued, they'd fail with "no handler registered."

- **Files:** Missing: `internal/fetcher/`, `internal/extractor/`
- **Severity:** No fetching means no reader. This is the next implementation milestone, not a defect per se — but the design promises a security model that doesn't exist yet. If feed URLs were accepted today, they'd be fetched by whatever ad-hoc HTTP code was written next, without the SSRF protections.

### 🟠 P1 — High-impact correctness/interop

**P1-1: Benchmark corpus is empty — blocks extraction decision**

`testdata/articles/manifest.json` contains `{"articles": []}`. The Phase 0 plan requires 20–50 articles for extraction candidate comparison. Without this, the extractor selection (go-readability vs trafilatura vs other) is blocked.

- **File:** `testdata/articles/manifest.json`

**P1-2: Domain models missing for all non-Job types**

The migration creates tables for `feeds`, `articles`, `article_content_versions`, `activity_events`, and `article_state`, but `internal/model/` contains only `job.go`. No Go domain types exist for any of the core entities.

- **Files:** `internal/model/job.go` (only), `internal/db/migrations/001_core.sql` (5 unmapped tables)
- **Impact:** Store implementations for feeds/articles/activity can't be written without these types.

**P1-3: Duplicate `generateID()` with predictable fallbacks**

Both `internal/store/jobs.go:36-43` and `internal/app/app.go:51-58` have identical ID generation functions. Both fall back to `"fallback-%d"` when `crypto/rand` fails — a predictable, non-cryptographic fallback. Extract to a shared utility in `internal/model` or `internal/id`.

**P1-4: Missing ADRs for extraction strategy, sync protocol, and error contract**

Three design decisions that affect the entire Phase 1 implementation surface are undocumented:
- **Extraction strategy:** How was the extraction adapter selected? What were the candidates and results?
- **Sync protocol:** Cursor semantics, conflict resolution, offline queue behavior, Room schema alignment.
- **Error contract:** The design mandates "stable JSON shape containing machine code, human message, retryability, and request ID" but no ADR formalizes this.

### 🟡 P2 — Meaningful quality/coherence

**P2-1: Android plan is under-specified compared to server plan**

The Phase 1 service plan is 10 detailed implementation slices. The Android workstream is a single paragraph. For a "phone-first" product, the Android surface (Room schema, Compose screens, WorkManager, offline queue, OPML import, encrypted token storage) deserves comparable detail.

- **File:** `docs/plans/phase-0-and-1-service.md` (Android workstream section)

**P2-2: Constraint error detection via string matching is fragile**

`store/jobs.go:197-208` detects SQLite constraint violations by checking for substring matches in modernc error messages. If the driver's error format changes in an update, race-condition recovery in `Enqueue()` silently breaks.

- **File:** `internal/store/jobs.go:197-208`
- **Mitigation:** The schema-level partial unique index on `jobs(dedupe_key)` provides defense-in-depth. But ideally use `errors.As` against a typed error when modernc provides one.

**P2-3: No request body size limits**

No `http.MaxBytesReader` or body-size middleware exists. When POST endpoints are added (sync events, OPML import), large bodies could cause memory exhaustion.

- **File:** No implementation — missing from `internal/app/app.go`

**P2-4: Health endpoint auth toggle not implemented**

DESIGN.md says healthz should be unauthenticated *"only when explicitly configured."* No config flag exists. When auth middleware is added, this toggle must be included.

**P2-5: No structured error JSON format**

The `writeJSON` helper exists (`app.go:61-69`) but no error-response types or format are defined. Errors from future endpoints will have inconsistent shapes.

### 🔵 P3 — Polish/future-facing

**P3-1: Request-ID echo without validation**

`withRequestID` accepts arbitrary client-supplied `X-Request-ID` values without length or character validation. Enforce max length (e.g., 128 chars, ASCII printable only).

- **File:** `internal/app/app.go:41-43`

**P3-2: No rate limiting**

Acceptable for single-user Tailscale v0.1, but worth planning for before any public endpoints exist.

**P3-3: No startup warning for non-loopback plain-HTTP bind**

If `RAVEN_ADDR=0.0.0.0:8789`, traffic is unencrypted. A startup log warning would help operators catch misconfiguration.

**P3-4: Logging discipline needs proactive enforcement**

CLAUDE.md non-negotiable #9 says: *"Do not log tokens, raw article bodies, or chat content in normal logs."* Currently `slog.Info`/`slog.Error` are used directly with no redaction wrapper.

**P3-5: Backup SQL uses string interpolation** (operator-controlled, low risk)

`db.go:178` constructs `VACUUM INTO '<escaped>'` via `fmt.Sprintf`. Escaping is correct (single-quote doubling), but the path comes from operator configuration, not user input.

## 6. Aesthetic / Product Thesis

**Raven wants to be a librarian, not an editor.** This is the defining aesthetic, and it is oppositional: it positions itself against Kagi News, Artifact, Bulletin, and the broader "AI replaces your feed" trend.

**What this code values (visible in the implementation):**

1. **Evidence over inference.** Append-only events. Content hashes. Extraction provenance. The code is built to *know* what happened, not guess.

2. **"Boring" as a virtue.** SQLite, not Postgres. One binary, not microservices. Stdlib `net/http`, not Gin/Chi. No ORM. The `MaxOpenConns=1` rationale comment in `db.go:38-41` is classic: *"Limit the pool to one persistent connection so that connection-scoped PRAGMAs apply to every operation."*

3. **Durability as respect for data.** Jobs survive restarts. Leases expire and are reclaimed. Errors are stored, not swallowed. The `dead` status exists as a permanent record of failure — nothing is silently lost.

4. **Transparency as product feature.** The interest profile's provenance field ("you set this," "learned from 14 skips," "you said this in chat") turns model explainability into user-facing value.

**The thesis is coherent and defensible.** The question is whether the implementation velocity can sustain it through the remaining ~80% of v0.1.
